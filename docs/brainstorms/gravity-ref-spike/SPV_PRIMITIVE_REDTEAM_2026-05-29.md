## 1. Executive Summary

The SPV primitive's **cryptographic core is sound**: PoW-per-header, chain-linkage, chain-anchor binding, the 64-byte Merkle-forgery guard, the txid↔raw-tx bind, and merkle-depth binding are all present and correct (source-verified in `pow.py`, `chain.py`, `merkle.py`, `proof.py`). The most important takeaway is a **structural one, not a single exploit**: the Python verifier deliberately enforces *no* network difficulty — `verify_header_pow` checks each header only against *its own* nBits, there is no difficulty floor, no `expected_nbits`, and no cumulative-work selection. **For the current code this is contained**, because the only production consumer (the now-deprecated Gravity SPV-oracle swap) settles against an on-chain covenant whose ASM pins `nBits ∈ {expectedNBits, expectedNBitsNext}` — the covenant is the real gate and Python is strictly weaker (harmless here). **But the same primitive is explicitly RETAINED for one-way bridge-in/oracle/gate uses that have no covenant backstop and no counterparty refund**; the instant any such consumer ships, the no-difficulty gap becomes a cheap, real proof-forgery (CRITICAL). The genuinely *exploitable today* issues are a small cluster of trust-boundary bugs reachable when an attacker influences the BTC data source (the documented default for "dust"): a bypassable coinbase guard, single-source confirmation-depth inflation, an unbound merkle-block↔header binding, and a misleading "quorum is the forge defense" threat-model claim. **No fund-theft path exists in shipped code today, but the primitive must not be wired as a sole authority until `expected_nbits` enforcement + a difficulty floor land.**

**F-04 — Coinbase-as-payment guard (`pos==0`) bypassable via high bits of `pos`.** `CONFIRMED_EXPLOITABLE` (3/3 real; HIGH; two reviewers reproduced end-to-end).
- **File:** `src/pyrxd/spv/proof.py:212-215`; `src/pyrxd/spv/merkle.py:55, 135-136`.
- **Attack:** The guard is exact equality `if pos == 0: raise`, but the merkle path is derived from only the low `depth` bits: `build_branch` computes `direction = (pos >> i) & 1` for `i in 0..depth-1` (merkle.py:55, **source-confirmed**) with **no** `pos < 2**depth` check. Any `pos` that is a nonzero multiple of `2**depth` (e.g. `pos=2` at depth 1, `pos=8` at depth 3) produces the identical all-left branch as `pos=0` and matches the real coinbase→root path, yet sails past `pos==0`. Reviewers reproduced: `build()` rejected `pos=0` but RETURNED a valid `SpvProof` for `pos=2`. The covenant has **no** coinbase guard at all (no `pos` integer in its branch walk), so neither layer robustly excludes the coinbase.

- **Fix:** In both `build()` (proof.py ~212) and `verify_tx_in_block` (merkle.py ~135): reject `pos >= 2**branch_depth` AND reject the all-left coinbase path `(pos & (2**branch_depth - 1)) == 0`, not just `pos==0`. Best: identify the coinbase *structurally* — reject any tx whose first input prevout is the null outpoint (txid == 32 zero bytes, vout == 0xffffffff) — so it holds independent of `pos`. Tighten `build_branch` to raise when `pos >> len(merkle_be) != 0`. Add regression tests for `pos ∈ {0,2,4,8}` at depths 1–3.

## 4. REFUTED / Non-Issues
*(Checked thoroughly; documented so the audit's negative space is explicit.)*

- **F-07 — `ChainTracker.is_valid_root` no-PoW (filed HIGH) → REFUTED (0/3).** The function genuinely does only `header[36:68] == merkle_root` with no PoW/anchor (source-confirmed by reviewers), **but it is dead/library code**: `MerklePath.verify`/`is_valid_root` have zero production callers; the live SPV value gate is `SpvProofBuilder.build()` (which runs PoW + anchor + depth binding). `MerklePath` is used only for BEEF wallet-tx (de)serialization. Latent footgun, not a reachable bug. *Hardening:* route any future gate through `SpvProofBuilder`; add a docstring/guard that this path is NOT a value gate.
- **F-08 — `SpvProof` construction guard bypassable via `dataclasses.replace`/`copy.copy` (filed HIGH) → REFUTED (0/3).** The `_token` field is preserved by `replace`/`copy` (mechanically true), **but no trust boundary is crossed**: `SpvProof` is built in-process from the taker's own data, never deserialized from a counterparty; `_BUILDER_TOKEN` is freely importable, so in-process code is already omnipotent; and the on-chain covenant independently re-verifies the embedded bytes. The docstring's "only via `build()`" overclaim is the only real defect. *Hardening:* make `_token` unstorable (thread-local one-shot in `build()`); fix the docstring.
- **F-09 — C-PARSER-2 fix lives only in `build()`, not in public `verify_payment` (filed HIGH) → CONFIRMED_HARDENING-LOW (1/3).** Standalone `verify_payment` does accept a payment-shaped blob at a scriptSig offset (reproduced), **but it has no standalone production caller** (`build()` is the only one, and it gates with `_output_offsets` one line prior), and the covenant is structurally immune (self-walks outputs, no caller-supplied offset). A real unsafe-by-default public API for *future* oracle/bridge use. *Hardening:* move the boundary/membership check into `verify_payment`, or demote it from the public `__all__`.
- **F-13 — Merkle sentinel-byte divergence (filed HIGH) → REFUTED (0/3).** See §2 LOW entry — sentinel padding never reaches Python's `compute_root`; both layers agree on the real branch.
- **F-14 — No duplicate-node (sibling==current) rejection / CVE-2012-2459 (filed LOW) → REFUTED (0/3).** The missing `sibling != current` check is real, but the merkle root is pinned by a PoW- and anchor-verified header, so self-pairing cannot reach the genuine root (reviewers proved by PoC); arbitrary-tx injection is blocked by the `len > 64` guard + txid bind. Optional defense-in-depth only.
- **F-19 — `merkle_path.find_or_compute_leaf` unmemoized recursion CPU DoS (filed MEDIUM) → CONFIRMED_HARDENING-LOW (1/3).** The quadratic/exponential blowup is real (reproduced), **but `from_beef`/`MerklePath.from_reader` have zero production callers** and the SPV value path uses the iterative `spv/merkle.py`. *Hardening:* cap `tree_height` (~≤32), memoize, validate counts before recursion — before any BEEF ingestion ships.
- **F-20 — Untyped exceptions escape from `merkle_path.py` (filed MEDIUM) → REFUTED (0/3).** Raw `TypeError`/`ValueError` leaks are real, but the cited `Transaction.from_hex → MerklePath.from_reader` chain is wrong (`from_hex` never builds a `MerklePath` and suppresses all exceptions), and `from_beef` has no production callers. Gravity/spv use the already-hardened `spv/merkle.py`. *Hardening:* mirror the typed-error discipline before wiring `from_beef`.
- **F-21 — `electrumx.get_transaction_merkle` builds a BUMP-inconsistent path (filed MEDIUM) → REFUTED (0/3).** Builder is genuinely broken for multi-tx blocks (fail-closed, never forges), in a method with zero callers, off the SPV value path. *Hardening:* delete or fix it.
- **F-22 — `electrumx` merkle handler computes/discards `2**len` + O(N) list (filed LOW) → CONFIRMED_HARDENING-INFO (1/3).** Dead `2**len` line is real; the DoS is bounded (1 MiB websockets frame cap; duplicate-offset guard short-circuits) and the method has no production caller. *Hardening:* delete the dead line; reject implausibly long merkle lists (>~40).
- **F-23 — `compute_root()` with no txid validates an arbitrary first-leaf branch (filed LOW) → REFUTED (0/3).** `MerklePath.__init__` already asserts all level-0 leaves yield the same root (proven), and the BEEF path is not on the SPV value boundary. Docstring overstates the risk.
- **F-24 — `CovenantParams` stores a `bytearray` without copying (TOCTOU, filed MEDIUM) → CONFIRMED_HARDENING-INFO (1/3).** Shared-reference mutation reproduced (source-confirmed: proof.py:127/140 store the reference uncopied), **but no production caller passes a bytearray, no code re-reads `covenant_params` after `build()`, and `build()` is synchronous (no mutation window)**. *Hardening:* copy to immutable `bytes` in `__post_init__` (or use the `Hex20`/`Hex32` newtypes).
